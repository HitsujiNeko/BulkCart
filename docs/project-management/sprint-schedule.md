## 🔄 Sprint 運用ルール

### Sprint Planning（スプリント開始時）

**日時**: 各スプリント初日 10:00-11:00

**アジェンダ**:
1. 前スプリントの振り返り
2. 今スプリントのゴール確認
3. Issue の Priority 見直し
4. タスクの工数見積もり
5. Sprint Backlog 確定

### Daily Standup（毎日）

**日時**: 毎日 10:00-10:15（15分）

**3つの質問**:
1. 昨日やったこと
2. 今日やること
3. ブロッカーはあるか

### Sprint Review（スプリント最終日）

**日時**: 毎金曜日 17:00-18:00

**アジェンダ**:
1. 完了 Issue のデモ
2. 未完了 Issue の確認
3. ステークホルダーフィードバック
4. 次スプリントへの持ち越し判断

### Sprint Retrospective（振り返り）

**日時**: 毎金曜日 18:00-18:30

**アジェンダ**:
1. **Keep**: 良かったこと
2. **Problem**: 問題点
3. **Try**: 次に試すこと

## 📊 進捗管理指標

### Burndown Chart

各スプリントで以下を記録:

- 残タスク数（Issue ベース）
- 残工数（時間ベース）
- Velocity（週あたりの完了 Issue 数）

### Quality Metrics

- **Code Coverage**: >80%
- **E2E Test Pass Rate**: 100%
- **Lighthouse Score**: >80
- **API Response Time**: <200ms
- **Bug Escape Rate**: <5%

---

## 🚨 リスク管理

### 想定リスクと対策

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| Supabase 無料枠超過 | High | Low | 早期アラート設定、Proプラン準備 |
| API レスポンス遅延 | Medium | Medium | Redis キャッシュ導入、クエリ最適化 |
| 献立生成ロジック不備 | High | Medium | Unit Test 充実、β テスト慎重実施 |
| 外部サービス障害 | High | Low | フォールバック UI、エラーハンドリング |
| スケジュール遅延 | Medium | Medium | バッファ週を設定、MVP スコープ調整 |
